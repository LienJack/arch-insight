# Agent記憶システムの設計哲学：5つのパラダイム比較から抽出可能なパターン

> エンジニア/アーキテクト向けの「設計参考解説」であり、製品比較レビューではありません。
> 分析フレームワークは Claude Code ContextManage の7次元モデル（Visibility / Authority / Temperature / Shape / Retrieval / Compression / Boundary）を採用。
> 5つのリファレンスリポジトリはすべて 2026-05-06 の main ブランチに固定。

---

## 1. 読む価値：Agent記憶はもはや「やるかやらないか」の問題ではなく、「どうやれば間違えないか」の問題

Agentシステムを設計しているなら、いずれこの問題に直面します：

> Agentの呼び出しは毎回ステートレスです。ユーザーが何を言ったか、前回どのファイルを読んだか、テストがなぜ失敗したか——これらの情報が永続化されなければ、Agentは毎回ゼロから推測し直さなければなりません。

過去2年間、この問題を解決するオープンソースプロジェクトが急速に登場しました。mem0（54.9K stars）、Graphiti（25.7K stars）、Letta（22.5K stars）、LangMem（1.4K stars）、Basic Memory（3K stars）——5つのプロジェクト、5つのまったく異なる「記憶」の定義。

比較なしでは、2つの極端に陥りやすいです：

- 「ベクターDB + RAGで十分」——しかし関係クエリができず、時間推論ができず、競合解決ができないことに気づく。
- 「知識グラフにしよう」——しかしグラフDBの維持コストが記憶システム自体の価値をはるかに超えることに気づく。

この記事の目的は、5つのプロジェクトの中から、自前のAgent記憶システムを構築するための最も安定したエンジニアリングパスを見つけることです——誰かを丸写しするのではなく、それぞれのトレードオフを理解し、自分自身の選択をすることです。

**5つのリファレンスリポジトリ：**

| プロジェクト | ポジショニング | 記憶モデル | 検索方式 | ライセンス |
|-------------|---------------|-----------|---------|-----------|
| [mem0](https://github.com/mem0ai/mem0) | 汎用AI Agent記憶層 | ベクトル化された事実断片 | セマンティック + BM25 + エンティティリンキング | Apache 2.0 |
| [Graphiti](https://github.com/getzep/graphiti) | 時系列知識グラフ記憶エンジン | Node/Edge + 時間ウィンドウ | セマンティック + BM25 + グラフ探索 | Apache 2.0 |
| [Letta](https://github.com/letta-ai/letta) | ステートフルAgentプラットフォーム | 構造化メモリブロック | コンテキストコンパイル型注入 | Apache 2.0 |
| [LangMem](https://github.com/langchain-ai/langmem) | Agent長期記憶ツール | Storeエントリ + Tool | ベクトル検索（LangGraph Store） | MIT |
| [Basic Memory](https://github.com/basicmachines-co/basic-memory) | ローカルファースト知識ベース記憶 | Markdownファイル + SQLite | FTS5 + FastEmbed ベクトル | AGPLv3 |

## 2. まず要件を合わせる：異なる記憶システムは異なる問題を解決する

技術的詳細に入る前に、最も重要な判断を下しましょう：

> **必要なのは「最高の記憶システム」ではなく、「Agentの形態に合った記憶システム」です。**

5つのパラダイムの核心的な違いは、モデル選択やAPI設計ではなく、より根本的な問題にあります：

**「記憶」はどのような物理的形態をとるべきか、そしていつモデルの視野に入るべきか？**

### 2.1 検索型 vs コンパイル型：モデルへの記憶注入の分水嶺

これが5つのシステムの最上位の分岐点です。

**検索型**：記憶は外部に保存され、Agentが必要な時に検索インターフェースを呼び出し、結果をプロンプトにどれだけ注入するか自分で決定します。
- mem0：`memory.search(query)` → `{"results": [...]}` を返す
- Graphiti：`graphiti.search(query)` → `list[EntityEdge]` を返す
- LangMem：`search_memory(query)` → 記憶エントリを返す
- Basic Memory：`read_note(identifier)` → Markdown内容を返す

**コンパイル型**：毎回のモデル呼び出し前にシステムが自動的に記憶をコンテキストにコンパイルします。Agentは検索判断をする必要がありません。
- Letta：`Memory.compile()` → Core Memoryブロックを直接XMLにレンダリングしてシステムプロンプトに注入

この分水嶺は4つの重要な結果を決定します：

| 次元 | 検索型（mem0/Graphiti/LangMem/Basic Memory） | コンパイル型（Letta） |
|-------|----------------------------------------------|----------------------|
| Agentの自律性 | 高い——Agentがいつ、何を、どれだけ注入するかを自分で決定 | 低い——システムが毎回コア記憶を確実に表示 |
| 記憶規模の上限 | 高い——数百万件に到達可能、検索で圧縮 | 低い——コンテキストウィンドウのトークン数に制限 |
| 検索失敗リスク | あり——Agentが検索を忘れるか、結果が不十分 | なし——コア記憶は決して失われない |
| コンテキスト汚染リスク | 低い——Agentに選択権がある | 中——古い記憶が更新されなければコンテキストに残り続ける |

明確なエンジニアリング結論：**長いタスクと高密度なツール結果を持つプログラミングCLI Agentには検索型が適しており、「あなたが誰かを知る」必要のあるチャットコンパニオンやカスタマーサービスにはコンパイル型が適しています。** 汎用Agentプラットフォームを構築する場合は両方必要です——Lettaがこれを証明しています（Core Memoryはコンパイル型、Archival Memoryは検索型）。

### 2.2 5つのストレージモデルの決定的な影響

各ストレージモデルは、検索能力の境界を直接決定します。

```
ストレージモデル → できるクエリ → 適さないクエリ
```

**mem0：ベクトル + BM25 + エンティティリンキング**
- ✓ セマンティック類似度クエリ（「ユーザーはどんなエディターが好き？」→「vimを好む」を返す）
- ✓ キーワード完全一致（「vim」→ BM25ヒット）
- ✓ エンティティ関連付け（「Alice」→ Aliceに言及するすべての記憶を検索）
- ✗ 関係クエリ（「誰がAliceと一緒に働いている？」——ベクトルでは不可）
- ✗ 時間推論（「AliceはVSCodeに切り替える前、どんなエディターを使っていた？」——mem0に事実のタイムスタンプなし）

**Graphiti：時系列知識グラフ**
- ✓ 関係クエリ（`(Alice)-[WORKS_WITH]->(Bob)` グラフ探索）
- ✓ 時間推論（「2025年のこの事実は何だった？」有効性ウィンドウクエリ）
- ✓ 来歴（「この結論はどの会話から来た？」Episode → Edge provenance）
- ✗ メンテナンスコスト（グラフDB Neo4j/FalkorDB/Kuzuが必要）
- ✗ あいまいセマンティッククエリは純ベクトル解決策ほど直接的でない（マルチパス融合が必要）

**Letta：構造化ブロック + Archivalベクトル検索**
- ✓ 正確なコンテキスト制御（`human`ブロックに書かれた内容がそのまま表示される）
- ✓ 明確な階層化（Core Memory vs Archival Memory vs Conversation Memory）
- ✗ 検索能力が弱い（Archival検索は単純なテキストマッチングに依存）
- ✗ ブロック数に制限（各ブロックに文字数上限 `CORE_MEMORY_BLOCK_CHAR_LIMIT`）

**LangMem：LangGraph BaseStore（キー値 + ベクトルインデックス）**
- ✓ エコシステム統合（LangGraphにゼロ設定でアクセス）
- ✓ ホットパス/バックグラウンド分離（Agentの能動的管理 + バックグラウンド自動抽出）
- ✗ エコシステムロックイン（LangChain/LangGraph以外のプロジェクトでは使用不可）
- ✗ カスタム検索が弱い（Storeの組み込みインデックスに依存）

**Basic Memory：Markdownファイル + SQLite（FTS5 + FastEmbed）**
- ✓ 人間が読める（Obsidianで直接開いて編集可能）
- ✓ ローカル制御（データは自分のファイルシステム上にある）
- ✓ ゼロインフラストラクチャ（グラフDB、ベクトルDB不要）
- ✗ 構造化クエリが弱い（複雑な関係クエリは不可）
- ✗ 規模の上限（ファイルシステム + SQLiteは数百万エンティティに不向き）

**核心推論**：Agentが主に事実QA（「ユーザーは何が好き？」）を行うなら、ベクトルDBで十分です。因果関係クエリ（「なぜこのコードは変更された？以前誰が変更した？」）が必要なら、時系列グラフの方が適しています。Agentがローカル知識ベースの読み書きを必要とするなら、Markdown + MCPが最もシンプルで十分な解決策です。

## 3. メインフロー分解：5つのシステムが「記憶→検索」ループを完成させる方法

### 3.1 mem0：ADD-Only 3フェーズバッチ処理パイプライン

mem0 V3（2026年4月リリース）は、CRUDからADD-onlyへの根本的な設計転換を行いました。

**書き込みフロー**（`mem0/memory/main.py:573-660`）：

```
add(messages, user_id, agent_id, run_id)
  ├─ Phase 0: コンテキスト収集
  │   └─ SQLiteから最新10メッセージを取得 + 入力メッセージを解析
  │
  ├─ Phase 1: 既存記憶の検索（関連付け用）
  │   └─ embedding(query=messages) → vector_store.search(top_k=10)
  │
  ├─ Phase 2: LLMによる新事実抽出
  │   └─ ADDITIVE_EXTRACTION_PROMPT → LLM → 新記憶リスト
  │   └─ 要点：ADDのみ返し、UPDATE/DELETEは返さない
  │
  ├─ Phase 3: エンティティ抽出 + リンキング
  │   └─ extract_entities() → 記憶ごとのエンティティリスト → ベクトル化 → クロスメモリエンティティリンキング
  │
  ├─ Phase 4: ベクトル化 + 書き込み
  │   └─ embedding(memory_text) → vector_store.insert()
  │
  └─ Phase 5: SQLite履歴への書き込み
      └─ db.insert_history(messages, memories)
```

**検索フロー**（`mem0/memory/main.py:1126-1237`）：

```
search(query, filters={user_id/agent_id/run_id}, top_k=20)
  ├─ 1. フィルタルール検証（少なくとも1つのエンティティID）
  ├─ 2. セマンティック検索: embedding(query) → vector_store.search()
  ├─ 3. BM25キーワード検索: lemmatize(query) → BM25 scorer
  ├─ 4. エンティティ検索: entity_store.search() → リンクされた記憶を見つける
  ├─ 5. 3経路融合: score_and_rank(semantic, BM25, entity)
  └─ 6. オプションのリランク
```

**設計の妙**：

ADD-onlyはmem0 V3の最大の哲学的転換です。旧バージョンでは「この新しい会話は記憶を作成すべきか、既存の記憶を更新すべきか、古い記憶を削除すべきか」を判断する必要がありました——この判断自体にLLM呼び出しが必要で、エラーが発生しやすかったのです。V3の解決策：判断を放棄し、すべてを追記し、マルチシグナル検索とスコアラーの重み付けに依存して、Agentが最新で最も関連性の高い情報を確実に取得できるようにします。

これは次のように理解できます：**書き込み時の競合解決コストを、検索時のキュレーションコストに移行したのです。** 書き込みはシンプルで高スループットになり、検索がより重要になりました。

> **推論**：ADD-onlyは「事実が自然に蓄積され、古い事実が一般的に誤った情報にならない」シナリオ（ユーザー設定、プロジェクト規約など）に最適です。「事実が正確でなければならず、矛盾する可能性がある」シナリオ（システム設定、権限ルールなど）には適していません。

### 3.2 Graphiti：Episode → Node → Edge グラフ構築パイプライン

Graphitiの書き込み操作はテキストを保存するのではなく、会話（Episode）をグラフ構造に変換します。

**書き込みフロー**（`graphiti_core/graphiti.py:933-1180`）：

```
add_episode(name, episode_body, source_description, reference_time)
  ├─ 1. 先行エピソードの検索（コンテキスト提供）
  │   └─ retrieve_episodes(ref_time, last_n=RELEVANT_SCHEMA_LIMIT)
  │
  ├─ 2. エピソード本文からNode（エンティティ）を抽出
  │   └─ extract_nodes(episode, previous_episodes) → LLM → list[EntityNode]
  │
  ├─ 3. Nodeの重複排除 + 解決
  │   └─ resolve_extracted_nodes(nodes) → 既存Nodeにマージ、または新規Node作成
  │
  ├─ 4. エピソード本文からEdge（関係）を抽出
  │   └─ extract_edges(episode, resolved_nodes) → LLM → list[EntityEdge]
  │
  ├─ 5. Edge解決 + 事実期限切れ管理
  │   └─ resolve_extracted_edge(edge) → 新事実が既存事実と競合する場合、旧事実を無効化
  │
  ├─ 6. EpisodicEdgeの構築（Episode ← → Entity の関連エッジ）
  │   └─ build_episodic_edges(episode, nodes, edges)
  │
  └─ 7. オプション：コミュニティサマリーの更新
      └─ update_community(nodes, edges)
```

**検索フロー**（`graphiti_core/graphiti.py:1550-1567`）：

```
search_(query, config=COMBINED_HYBRID_SEARCH_CROSS_ENCODER)
  ├─ セマンティック検索: embedding(query) → 関連EntityEdgeを検索
  ├─ キーワード検索: BM25 → ノード名/エッジ種類/事実テキストをマッチ
  ├─ グラフ探索: 中心ノードからBFS → 関連事実を発見
  ├─ Cross-encoderリランク
  └─ SearchResults(nodes, edges, episodes) を返す
```

**設計の妙**：

Graphitiの最もユニークな抽象化は**事実有効性ウィンドウ**です。各EntityEdgeには `valid_at` と `invalid_at` フィールドがあります。新しい事実が古い事実と競合した場合（例：「Aliceは黄色が好きではなくなり、今は青が好き」）、システムは古い事実を削除せず、`("Alice", "LIKES", "yellow")` の `invalid_at` を現在時刻に設定します。クエリはデフォルトで現在有効な事実を返しますが、「2025年のAliceは何が好きだったか」のようなタイムトラベルクエリもサポートします。

> これは時系列データベースの設計パターンを知識グラフに移行したものです。代償として、すべての事実書き込みに競合検出（同じエンティティペア間に同じタイプのエッジが既に存在するか）が必要となり、LLM抽出フェーズに追加の複雑さをもたらします。

### 3.3 Letta：メモリブロックによるコンパイル型注入

Lettaの記憶モデルは最もシンプルで直接的です：**記憶とは、システムプロンプトに構造的にコンパイルされるブロックです。**

**核心抽象**（`letta/schemas/block.py:13-66`, `letta/schemas/memory.py:68-103`）：

```python
class BaseBlock(LettaBase):
    value: str           # 記憶内容
    limit: int           # 文字数上限 (CORE_MEMORY_BLOCK_CHAR_LIMIT)
    label: str           # 'human', 'persona', またはカスタム
    description: str     # ブロックの説明
    read_only: bool      # Agentが読み取りのみかどうか

class Memory(BaseModel):
    blocks: List[Block]  # Core Memoryブロック（毎回表示）
    file_blocks: List[FileBlock]  # 添付ファイル用の特殊ブロック

# デフォルトの2ブロック
DEFAULT_BLOCKS = [Human(value=""), Persona(value="")]
```

**書き込みと注入**（`letta/schemas/memory.py:688-732`, `letta/schemas/memory.py:804-837`）：

Agentはツール関数を通じてブロックを変更します：
```python
# core_memory_replace: ブロック内の指定内容を置換
# core_memory_append: ブロックに内容を追加
```

毎回のモデル呼び出し前に、`Memory.compile()` がブロックを次のようなXMLにレンダリングします：
```xml
<memory_blocks>
<human>
<description>ユーザーに関する事実</description>
<value>ユーザーはダークモードを好み、vimを使用</value>
</human>
<persona>
<description>Agentの自己認識</description>
<value>私はプログラミングアシスタントです</value>
</persona>
</memory_blocks>
```

**設計の妙**：

Lettaの `human`/`persona` 分離は古典的な設計です——**「ユーザーが誰か」と「私が誰か」を異なるコンテキスト領域に分離します。** 7次元モデルのVisibility次元において、これによりAgentは外部事実（ユーザー設定）と内部制約（自身の振る舞い方）を明確に区別できます。

さらに重要なことに、Lettaはブロックに `limit`（文字数上限）を課しています——これは見落とされがちですが、実際には重要な制約です。上限のないブロックは最終的にコンテキストウィンドウを埋め尽くすまで成長しますが、上限があれば、Agentは真に重要な長期記憶を選択的に保持しなければなりません。

> **保留中の検証**：LettaのArchival Memory検索能力（`letta/schemas/memory.py:868-884`）は専用検索システムより弱いように見えます。実際の使用でArchival検索の再現率が低い場合、AgentはCore Memoryブロックに過度に依存し、Core Memoryの `limit` への圧力が増大します。

### 3.4 LangMem：記憶管理をAgentのToolに変える

LangMemの設計は最もユニークです——記憶自体を保存するのではなく、Agentに記憶管理の「ツール」を提供します。

**核心抽象**（`src/langmem/knowledge/tools.py:25-305`）：

```python
create_manage_memory_tool(
    namespace=("memories", "{user_id}"),  # 実行時に解決
    actions_permitted=("create", "update", "delete"),
) -> StructuredTool

create_search_memory_tool(
    namespace=("memories", "{user_id}"),
) -> StructuredTool
```

**Agent視点での使用法**：

```
Agentが受信：「ダークモードが好きだと覚えておいて」
  → Agentが判断：この情報は保存が必要
  → Agentが manage_memory(action="create", content="ユーザーはダークモードを好む") を呼び出し
  → 記憶がLangGraph Storeに書き込まれる

Agentが受信：「私のインターフェース設定は？」
  → Agentが判断：記憶の検索が必要
  → Agentが search_memory(query="インターフェース設定") を呼び出し
  → 結果を取得：「ユーザーはダークモードを好む」
```

**バックグラウンドモード**（`src/langmem/knowledge/extraction.py:113-447`）：

```
Background MemoryManager
  ├─ 会話スレッドを監視
  ├─ 記憶を自動抽出 (create_thread_extractor)
  ├─ 既存記憶をマージ/更新（セマンティック重複排除）
  ├─ ユーザープロファイルを更新 (update_profile)
  └─ プロンプトを最適化 (optimize_prompt)
```

**設計の妙**：

LangMemは記憶操作の決定権を完全にAgent（ホットパス）またはバックグラウンドマネージャー（バックグラウンド）に委ねます。これにより、mem0の設計上の課題を解決しています：**「誰がいつ記憶を書き込むかを決定するのか？」** mem0は `add()` が呼ばれるたびに書き込みます——つまり、呼び出し元（Agentフレームワーク）がいつ `add()` を呼び出すべきかを判断しなければなりません。LangMemはツールパターンを使用します：Agent自身が `read_file` をいつ呼び出すべきかを知っているのと同様に、`manage_memory` をいつ呼び出すべきかを知っています。

しかし、これにはリスクもあります：**Agentが `manage_memory` を呼び出すのを忘れると、重要な情報が失われます。** Background MemoryManagerはこのパスへのパッチです——Agentの能動的な呼び出しに依存せず、会話から自動的に記憶を抽出しマージします。

> **推論**：LangMemのホットパス + バックグラウンドの二重パス設計は「人間の記憶」モデルに最も近い——一部の記憶は能動的に記録され（ホットパス）、一部は事後に自動的に抽出されます（バックグラウンド）。ただし、実装の複雑さは無視できません：バックグラウンドマネージャーは自動抽出された記憶とAgentが能動的に記録した記憶の間の競合を処理する必要があります。

### 3.5 Basic Memory：ファイル＝記憶

Basic Memoryの哲学は最も直感に反しますが、最も実用的です：**記憶は人間とAIの両方が読み書きできるMarkdownファイルに存在します。**

**核心抽象**（`src/basic_memory/mcp/tools/build_context.py:136-166`）：

```
ベースパス: ~/basic-memory/
ファイル構造:
  ├── project-a/
  │   ├── notes/
  │   │   ├── coffee-brewing.md
  │   │   └── vim-config.md
  │   └── schema/
  └── project-b/

MCPツール:
  ├── write_note    → Markdownファイルを作成/上書き
  ├── read_note     → Markdownファイルを読み取り
  ├── edit_note     → 内容を追加/置換
  ├── delete_note   → ファイルを削除
  ├── search_notes  → FTS5 + FastEmbed ハイブリッド検索
  ├── build_context → wikiリンクを解析してグラフコンテキストを構築
  └── list_directory → ファイルツリーの閲覧
```

**build_contextの動作**：

```
Agentが「coffee」について知りたい
  → build_context(uri="coffee")
  → SQLiteを検索して coffee-brewing.md を発見
  → Markdown frontmatterを解析（title, tags, permalink）
  → [[wiki links]] を解析 → 関連ノートを発見
  → GraphContext(results=[primary + observations + relations]) を構築
  → 構造化されたMarkdownを返す
```

**設計の妙**：

Basic Memoryは賢い簡略化を行っています：エンティティ抽出、事実競合検出、複雑なグラフクエリを避けています。2つのものに依存しています：
1. **人間が書くMarkdownはすでに十分に構造化されている**（frontmatter + wiki links + observations/relations パターン）
2. **MCPプロトコルにより、AIも人間と同じようにノートを開き、読み、検索できる**

これにより、5つのプロジェクトの中で最も複雑な部分——LLM抽出パイプライン——を回避しています。代償は、ユーザーが積極的にノートを書かないか、ノートが構造化されていない場合、システムの記憶能力が弱まることです。

> このトレードオフは注目に値します：「技術的に不可能」ではなく「エンジニアリングとして選択しなかった」のです。個人の知識ベース記憶にはこの選択は賢明ですが、会話から自動的に記憶を抽出する必要がある商用Agentには、mem0やLangMemの自動抽出が必要です。

## 4. 7次元モデル横断比較：「機能比較」から「設計トレードオフ分析」へ

コンテキスト管理の7次元：Visibility、Authority、Temperature、Shape、Retrieval、Compression、Boundary。ここでは5つのシステムを同じ座標系に配置します。

### 4.1 Visibility：記憶ストレージとモデル可視性の分離

5つのシステムに共通するパターン：**記憶ストレージ ≠ モデルコンテキスト。** 記憶システムは保存と検索を行い、Agentが何を注入するかを決定します。

唯一の例外はLettaのCore Memoryです——ブロックを直接コンテキストウィンドウにレンダリングします。つまり、Lettaのコア記憶はAgentが検索判断をする必要がありませんが、記憶規模がコンテキストウィンドウによって制限されることを意味します。

### 4.2 Authority：競合解決は概ね委任されている

7次元モデルで最も成熟度の低い次元です。5つのシステムのいずれも、明示的な権威チェーン（System > Org > Project > User > Task State > Memory > Summary）を実装していません。

実際の分岐点は**競合が誰に委任されるか**です：
- **時間（Graphiti）**：事実は自動的に期限切れになり、時間が最も権威ある裁定者
- **Agent（mem0 ADD-only + LangMem hot path）**：Agentが検索結果に基づいてどちらが「真実」かを判断
- **ユーザー/Agentの明示的操作（Letta + Basic Memory）**：最後の書き込みが優先

> **推論**：MVP段階の自構築システムでは、「時間ベースの裁定 + Agentによる上書き機能」が妥当な出発点です。Graphitiは時間をデータモデルに組み込んだ唯一のシステムであり、他の4つはアプリケーション層で手動の情報鮮度管理が必要です。

### 4.3 Temperature：ホット/コールド階層化は普遍的に存在するが明示的ではない

5つのシステムすべてにホット/コールドの概念がありますが、実装方法は異なります：
- **Graphitiが最も明示的**：事実の `valid_at`/`invalid_at` フィールドは組み込みのホット→コールド遷移メカニズム
- **mem0**：`get_last_messages(limit=10)` で最近のテールを「ホット」に保持、ベクトル検索の残りは「ウォーム」
- **Letta**：Core Memoryは「ホット」、Archival検索結果は「ウォーム」
- **LangMem**：ホットパス書き込みは「ホット」、バックグラウンド抽出は「ウォーム」
- **Basic Memory**：現在開いているノートは「ホット」、検索されたノートは「ウォーム」

**主要な観察**：完全なTemperature Managerを持つシステムはありません——温度遷移は（Graphitiを除いて）暗黙的です。これは、自構築システムにとって**Temperature Managerが差別化の機会である**ことを意味します：Hot → Warm → Cold → Frozen の遷移ルールを明示的に管理することで、コンテキスト汚染を大幅に削減できます。

### 4.4 Shape：誰もがテキスト→構造化への形態変換を行っている

5つのシステムに共通するパターン：
```
生の入力（会話/ドキュメント）
  → LLMが構造化事実を抽出
  → 構造化表現を保存
  → 検索時にAgentに最も適した形式で返す
```

Graphitiの形態変換が最も完全（テキスト → Node/Edge → Community summary → 時系列事実グラフ）ですが、最も重厚でもあります。mem0のADD-only抽出は最も軽量です。Basic Memoryは形態変換を行わないことを選択しています——Markdownをそのまま保存し、そのまま返します。

**明確なエンジニアリング提案**：形態変換のコスト（LLM呼び出し + スキーマ設計 + 競合処理）は便益（検索精度、関係クエリ能力）に比例します。mem0のADD-only抽出から始めて、徐々にエンティティリンキングとグラフ構造を追加する方が、最初からGraphitiの完全なパイプラインを使うよりもリスクが低いです。

### 4.5 Retrieval：マルチパス検索は必須だが、すべてを実装する必要はない

mem0とGraphitiの最先端の実践が証明しています：**セマンティック + キーワード + グラフ/エンティティの3経路融合が最良の検索パターンです。** ただしMVPの場合：
- **セマンティックベクトル検索**が最も代替不可（必須）
- **キーワード/BM25**はコストが非常に低く、推奨（mem0とGraphitiの両方が使用）
- **エンティティリンキング**はmem0 V3の新機能で、明確なエンティティドメインに適する（ユーザー設定、プロジェクト構成）
- **グラフ探索**はGraphitiの核心的な差別化要因であり、関係クエリが必要な場合にのみ投資する価値がある

### 4.6 Compression：目標は「引き継ぎメモ」であり、「読書感想文」ではない

5つのシステムすべてが同じパターンで圧縮しています：
```
大量の生情報 → LLM → 少量の構造化事実
```

ただし、圧縮目標の品質基準は異なります：
- **mem0** の ADDITIVE_EXTRACTION_PROMPT は、LLMに会話から独立したクリーンな事実文を抽出させる
- **LangMem** の `create_thread_extractor` はスレッド要約 + 記憶抽出を行う
- **Graphiti** の抽出はエンティティと関係を抽出し、要約は行わない

**重要なエンジニアリング判断**：圧縮は「何が起こったか」だけでなく、「どこで止まっていて、次に何をすべきか」も保持すべきです。mem0もLangMemもこれを達成していません——抽出された事実は独立しており、「タスク状態」を保持していません。長期タスクAgent向けの自構築システムでは、圧縮要約に現在のフェーズ、試行済みアプローチ、失敗記録、次のステップを保持するよう特に注意する必要があります。

### 4.7 Boundary：最も過小評価されている次元

5つのシステムすべてでBoundaryの実装は非常に基础的です：
- **最良**：mem0の `(user_id, agent_id, run_id)` 3次元分離
- **最も柔軟**：LangMemの実行時名前空間注入
- **最も粗い**：Basic Memoryのプロジェクトレベル分離
- **最も弱い**：Graphitiのgroup_id（データベースレベルパーティショニング）

**サブAgentコンテキスト分離を実装しているシステムはありません**——すべてのサブAgentが同じ記憶空間を共有しています。マルチAgent協調を計画している場合、これは特に注意すべき設計ギャップです。

## 5. 設計トレードオフ：知っておくべき5つの選択

### トレードオフ 1：ADD-Only vs CRUD —— 記憶システムの根本的な哲学的分岐

```
            ADD-Only                          CRUD
書き込み複雑度  低（追記のみ）                      高（create/update/deleteの判断が必要）
検索負荷        高（スコアラーによるキュレーションに依存）  中（記憶は常に最新状態を表現）
一貫性          弱（古い事実は期限切れにならない）        強（古い事実は更新または削除される）
監査可能性      強（完全な書き込み履歴）                  弱（古い値は上書きされる）
適切なシナリオ  ユーザー設定、プロジェクト規約            構成ルール、権限状態
代表プロジェクト mem0 V3                                LangMem
```

**選択アドバイス**：記憶が主に「嗜好/習慣/経験」のような累積的情報を扱う場合、ADD-onlyの方が安定しています。「ルール/設定/権限」のような正確な情報を含む場合、CRUDが必要です。

### トレードオフ 2：時系列グラフ vs ベクトルDB —— ストレージモデルが検索能力の上限を決める

```
                      時系列グラフ (Graphiti)              ベクトルDB (mem0)
関係クエリ             ✓✓✓ (グラフ探索、圧倒的優位)         ✗ (パスクエリ不可)
時間推論               ✓✓✓ (有効性ウィンドウ)                ✗ (時間概念なし)
あいまいセマンティクス  ✓ (マルチパス融合)                    ✓✓✓ (自然な優位性)
エンティティ来歴       ✓✓✓ (Episode provenence)              ✓ (エンティティリンキング)
インフラコスト         高 (グラフDBが必要)                  中 (ベクトルDBが必要)
適切なシナリオ         因果推論が必要な複雑Agent            事実QA型Agent
```

Graphitiの能力は高いですが、インフラコストもはるかに高くなります。**ほとんどのAgentシナリオに完全なグラフDBは必要ありません**——mem0の「ベクトル + BM25 + エンティティリンキング」ですでに80%のシナリオをカバーしています。

### トレードオフ 3：エコシステムバインディングは両刃の剣

| プロジェクト | エコシステムバインディング度 | コスト |
|-------------|---------------------------|-------|
| LangMem | LangGraphに深く結合 | LangChain以外のプロジェクトでは使用不可 |
| Basic Memory | MCPに結合 | MCPクライアント以外では使用不可 |
| mem0 | 独立 | 統合ロジックを自分で書く必要あり |
| Graphiti | グラフDBに結合 | グラフDBの運用が必要 |
| Letta | 独立 | 自身のAgentランタイムに統合する必要あり |

**結論**：プロジェクトですでにLangGraphを使用している場合、LangMemのゼロ設定統合の価値はエコシステムロックインのコストを上回ります。新規プロジェクトの場合、mem0が最も統合と置換が容易です。

### トレードオフ 4：自動抽出 vs Agent能動的管理

```
                 自動抽出 (mem0 / LangMem Background)    Agent能動的管理 (LangMem Hot Path / Basic Memory)
信頼性           毎回必ず書き込みが発生                    Agentがツール呼び出しを忘れる可能性あり
品質             extraction promptの質に依存               Agentの判断力に依存
コスト           追加LLM呼び出し                          追加LLM呼び出しゼロ（Agentはすでに判断コンテキストを持つ）
タイミング       会話後に非同期処理                        会話中にリアルタイム処理
適切なAgentタイプ プログラミングAgent、長期タスクAgent      チャットAgent、軽量Agent
```

**アドバイス**：両方を組み合わせて使用します——自動抽出をベースライン保証（重要な情報を失わない）とし、Agent能動的管理を最適化パス（記憶品質の向上）とします。

## 6. 借用可能なパターンと適用条件

### パターン 1：mem0のマルチパス融合検索パイプライン

**適用対象**：記憶検索が必要なあらゆるAgentシステム

**核心アプローチ**：
```
セマンティック検索 + BM25キーワード + エンティティリンキング → 3経路並列スコアリング → 融合ランキング → オプションのリランク
```

**なぜ借用する価値があるか**：mem0 V3はLoCoMoベンチマークで71.4から91.6（+20ポイント）に向上し、マルチパス融合が核心的な貢献要因です。高精度シナリオでの単一経路検索は、「セマンティックミス → 記憶喪失」という障害モードを起こしやすいです。

**コピーすべきでない場合**：総記憶数 < 1000件。この規模では純粋なセマンティック検索で十分であり、マルチパス融合の複雑さには価値がありません。

**移行時に優先検証**：BM25のlemmatizationが中国語で効果的かどうか。mem0のBM25実装は英語のlemmatization（`mem0/utils/lemmatization.py`）に依存しており、中国語シナリオではトークナイザーの置換または補足が必要です。

### パターン 2：Graphitiの事実有効性ウィンドウ

**適用対象**：記憶内の事実が時間とともに変化する可能性があるシナリオ（ユーザー設定、プロジェクト構成、チーム役割）

**核心アプローチ**：
```
EntityEdge {
    valid_at: datetime,     # 事実が真になった時刻
    invalid_at: datetime,   # 事実が失効した時刻（Null許容）
}
# 新事実が旧事実と競合 → 旧事実 invalid_at = now()
# クエリはデフォルトで有効な事実のみを返すが、タイムトラベルもサポート
```

**なぜ借用する価値があるか**：これは「古い情報を削除せず、しかし古い情報に汚染されない」ための最もシンプルなメカニズムです。ADD-onlyに1層の事実ライフサイクル管理を追加し、CRUDに1層の監査可能性を追加します。

**コピーすべきでない場合**：記憶内の事実がほとんど変わらない場合（知識ベースドキュメントインデックスなど）。そのようなシナリオでは、有効性ウィンドウは純粋な追加複雑性です。

**移行時に優先検証**：競合検出のコスト。Graphitiは新しい各エッジと既存エッジの間で競合を検出する必要があり、この検出の複雑さはエンティティ密度とともに増加します。「同じエンティティペア + 同じエッジタイプ」の単純な検出から始め、完全なセマンティック競合検出に進む前に検証します。

### パターン 3：LettaのHuman/Personaブロック分離

**適用対象**：「ユーザーが誰か」と「自分がどう振る舞うべきか」を記憶する必要があるあらゆるAgent

**核心アプローチ**：
```
human block:   ユーザー事実（嗜好、習慣、背景）
persona block: Agentの自己認識（役割、制約、行動指針）
```

**なぜ借用する価値があるか**：この分離は、コンテキスト汚染の中で最も見えにくい形態の1つ——**「ユーザーが言ったこと」を「自分が従うべきルール」と誤解すること**——を解決します。単一のコンテキスト領域では、モデルはこれら2種類の情報を区別するのが困難です。異なるXMLタグに分離することで、セマンティックガイダンスを通じてモデルが正しく区別しやすくなります。

**移行時に優先検証**：ブロックの文字数上限設定。大きすぎるとフィルタリング機能を果たせず、小さすぎると重要な情報を失います。2000文字から開始し、10セッション観察して調整します。

### パターン 4：LangMemのホットパス + バックグラウンド二重パス

**適用対象**：長期的な会話から知識を蓄積しながら、対話中のリアルタイム記憶管理も必要とするAgent

**核心アプローチ**：
```
ホットパス:   Agentが会話中に manage_memory/search_memory ツールを呼び出す
バックグラウンド:  Background MemoryManagerが会話スレッドから自動的に記憶を抽出/マージ/更新
```

**なぜ借用する価値があるか**：これは「人間の記憶モデル」に最も近いアナロジーです——即時記憶（能動的記録）+ 長期定着（睡眠時の自動抽出）。マルチターン対話Agentにとって、この二重パスは「ユーザーが明示的に記憶を要求する」と「システムが会話から自動的に学習する」の両方のシナリオをカバーします。

**コピーすべきでない場合**：シングルターンタスクAgent（一度きりのバグ修正など）。「長期的蓄積」の必要性がなければ、バックグラウンドモードは純粋な無駄です。

**移行時に優先検証**：バックグラウンド抽出とホットパス書き込みの間の競合処理。自動抽出された記憶とAgentが能動的に記録した記憶が矛盾する場合、明確なマージ戦略が必要です。

### パターン 5：Basic Memoryの人間-AI共有記憶フォーマット

**適用対象**：人間とAIが協力して知識を維持する必要があるシナリオ

**核心アプローチ**：
```
Markdown + frontmatter + [[wiki links]] + observations/relations パターン
→ 人間はObsidianで編集
→ AIはMCPツールを通じて読み書き
→ SQLiteで同期インデックス
```

**なぜ借用する価値があるか**：AI記憶システムで最も見落とされがちな問題を解決します——**AIが記憶を書いたとき、人間はどうやって確認し修正すればよいのか？** 記憶が人間に読めないベクトル埋め込みになっていると、人間は記憶システムの最終的な制御権を失います。

**コピーすべきでない場合**：純粋な機械対機械のAgentシステム。そのようなシナリオでは、人間の可読性は優先度ではなく、検索性能が重要です。

## 7. 自前Agent記憶システム構築のエンジニアリングロードマップ

5つのシステムの設計トレードオフ分析に基づき、7次元モデルに沿った自構築パスは以下の通りです：

### v0.1 MVP：「保存」と「検索」をまず作る

```
Retrieval Router:
  └─ 単一経路セマンティックベクトル検索 (embedding + vector store)

Compression Engine:
  └─ LLM抽出: 会話 → 構造化事実
  └─ mem0のADDITIVE_EXTRACTION_PROMPTを参考

Visibility Filter:
  └─ 検索結果を直接コンテキストに注入（Agentが注入量を決定）

ストレージ: SQLite + ベクトル拡張、または Qdrant
```

**やらないこと**：エンティティリンキング、マルチパス融合、温度管理、競合解決、サブAgent分離。

**なぜこの順序から始めるか**：検索は記憶システムの最も中核的な価値です。良い検索なしには、その後のすべての最適化は絵に描いた餅です。

### v1.1：マルチパス検索 + 温度管理

```
Retrieval Router:
  ├─ セマンティック検索（既存）
  ├─ BM25キーワード（新規、非常に低コスト）
  └─ エンティティリンキング（新規、mem0 entity store設計を参考）

Temperature Manager:
  ├─ Hot: 最近Nターンの会話から生成された記憶
  ├─ Warm: 検索でヒットした履歴記憶
  ├─ Cold: ヒットしなかった古い記憶（削除せず、コンテキストに入れない）
  └─ Frozen: 完全な会話トランスクリプト（監査用、LLMコンテキストには入れない）
```

### v1.2：境界分離 + 記憶ライフサイクル

```
Boundary Controller:
  ├─ user_id 分離（必須）
  ├─ agent_id 分離（マルチAgentシナリオ）
  └─ run_id 分離（単一タスクローカル記憶）

Authority Resolver:
  ├─ 時間優先（Graphitiの有効性ウィンドウを参考）
  ├─ Agentによる上書き可能（LangMemのホットパスを参考）
  └─ 重要な制約はハードコードしてLLMに入れない（7次元モデルのAuthorityチェーンを参考）
```

### v1.3+：マルチAgentシナリオ + サブAgentコンテキスト分離

```
Boundary Controller拡張:
  ├─ サブAgent入力は狭く：タスク + アーティファクト参照
  ├─ サブAgent出力は狭く：結論 + 証拠 + 信頼度
  └─ 構造化ハンドオフ（完全な会話履歴は渡さない）
```

**優先順位の原則**：各ステップで前のステップの必要性を検証してから投資します。v0.1でグラフDBを構築せず、マルチAgent要件なしにサブAgent分離を構築しません。

---

## 8. 一言でまとめると

> **Agent記憶システム選定の核心的な判断は「どれがより強力か」ではなく、「あなたのAgentが何の情報を失うことを最も恐れているか」です——嗜好事実を失うことを恐れるならベクトル検索（mem0パターン）、因果関係を失うことを恐れるなら時系列グラフ（Graphitiパターン）、コンテキストの正確な制御を失うことを恐れるなら構造化ブロック（Lettaパターン）、エコシステム統合を失うことを恐れるならフレームワークバインディング（LangMemパターン）、人間の可読性を失うことを恐れるならファイル＝記憶（Basic Memoryパターン）です。**
