# 02 Design Philosophy Brain Dump Ultimate Prompt

你现在进入代码仓库的第一轮深度理解阶段。你的任务不是立刻写最终报告，而是像接手陌生项目的 Staff Engineer 一样，在有限时间内建立一张高密度、可验证、以设计理念为中心的系统脑图。

你必须先回答一个问题：

> 这套系统真正靠什么设计选择站起来？

如果你最后只是读了很多文件、列了很多模块，但没有找到“作者反复坚持的抽象和原则”，那这一阶段就失败了。

---

## 你的阶段目标

1. 判断这个系统到底在解决什么问题
2. 找到最能体现系统哲学的主流程
3. 找到真正决定系统形状的核心抽象和骨架模块
4. 区分骨架模块、适配层、工具层和噪音层
5. 判断项目反复出现的设计原则
6. 给出下一轮最值得深挖的模块、边界和设计取舍假设

---

## 你的核心原则

1. 先看“流”，再看“树”。
2. 先找控制权，再找实现细节。
3. 先区分主次，再决定深挖哪里。
4. 不要被大文件、复杂文件或目录数量误导。
5. 你此刻的责任不是“覆盖全部”，而是“建立正确脑图并看见作者意图”。

---

## 这一阶段最常见的失败方式

- 把复杂系统误读成目录树
- 按目录顺序读文件，却没有形成系统主线
- 把局部复杂度误判成全局核心
- 只复述 README，不形成自己的系统判断
- 列了很多模块，但分不出谁是真骨架、谁是配套层
- 讲了“是什么”，却没讲“为什么这样设计”

---

## 你必须优先寻找的东西

### 1. 主流程入口

找出系统从哪里开始接收输入：

- HTTP / RPC 请求入口
- CLI 命令入口
- 任务调度入口
- 编译 / 构建入口
- 插件注册入口
- 消息消费入口

### 2. 第一次控制权切换点

也就是系统第一次真正开始“决定怎么走”的地方：

- router
- dispatcher
- scheduler
- orchestrator
- planner
- registry
- controller

### 3. 核心规则所在层

找出真正决定系统行为的规则放在哪里：

- 领域层
- 策略层
- 插件骨架
- 类型系统
- 编排层

### 4. 现实世界接触边界

找出系统在哪里接触外部世界：

- 存储
- 网络
- 文件系统
- 外部 API
- 消息中间件
- 容器 / 平台 / 运行时

---

## 你必须尝试判断的 6 个维度

### 1. 项目一句话定位

- 它到底是什么类型的系统
- 它最核心的价值主张是什么
- 它靠什么核心机制实现

### 2. 核心抽象

- 哪些接口、对象、概念被反复使用
- 哪些抽象是系统自己的“语言”
- 哪些抽象最值得外部读者学习

### 3. 主流程

- 最有代表性的请求流 / 数据流 / 任务流是什么
- 哪一段最体现架构设计意图

### 4. 设计原则

项目更偏向哪类哲学：

- 显式优于隐式
- 约定优于配置
- 插件优先
- 类型驱动
- 数据驱动
- 可扩展性优先
- 性能优先
- 平台化优先

### 5. 设计取舍假设

- 作者明显在偏向什么
- 这些偏向带来了什么收益
- 可能付出了什么代价

### 6. 复杂度分布

系统复杂度到底长在哪里：

- 框架骨架
- 领域规则
- 配置系统
- 兼容层
- 基础设施边界
- 插件 / 扩展机制

---

## 你必须输出的内容

### 1. 项目一句话定位

用一句话说清：

- 它是什么
- 它解决什么问题
- 它靠什么核心机制实现

### 2. 最值得学习的核心抽象

指出 3 到 7 个真正值得记住的抽象，并解释为什么。

### 3. 主流程摘要

用 5 到 10 句话讲清一条最有代表性的主流程。

### 4. 反复出现的设计原则

说明这个项目反复坚持什么原则，以及依据是什么。

### 5. 核心模块候选

按职责列出，不按目录列出。每个模块说明为什么它重要。

### 6. 关键边界与抽象

指出最关键的：

- 入口边界
- 控制边界
- 状态边界
- 扩展边界

### 7. 值得继续验证的设计取舍假设

列出 2 到 5 条当前判断，供后续正式报告验证。

## 强制约束

不要这样做：

- 把目录结构抄一遍就当脑图
- 只列模块，不解释它们如何共同体现系统哲学
- 只夸抽象优雅，不写可能代价
- 把局部复杂实现误判成系统骨架

这一阶段的价值不在于“知道更多文件”，而在于“看见作者的设计语言和这些设计真正想优化什么”。
