# arch-insight

ソースコードリポジトリを**設計判断**に変換する Agent Skill — ドキュメントの書き直しではなく、設計理念の抽出を行います。

他のプロジェクトを研究する際に、設計意図を素早く理解し、借用可能なパターンを抽出し、トレードオフを明確にできます。しかも、毎回新しい会話でリポジトリ全体を再スキャンする必要はありません。

## なぜこの Skill が必要か

LLM で他の人のコードリポジトリを分析する際によく遭遇する問題：

- **トークンの浪費** — 毎回リポジトリ全体をスキャンし、コンテキストが急速に消費される
- **重複作業** — 新しい会話のたびにゼロから始め、スキャンしたコードを再度スキャン
- **浅い分析** — モジュールリストとフローチャートだけで、設計推論もコード証拠もない
- **スタイルの不一致** — AI のデフォルトの書き方が自分のスタイルと大きく異なる
- **ガイド不足** — AI に何を分析させるべきか分からず、AI も何を見るべきか分からない

`arch-insight` は、段階的なイントーク、フォーカスしたコンテキストパッキング、スタイル調整、そして再説明ではなく判断を促すプロンプトを通じて、これらの問題を解決します。

## 使う場面

- オープンソースプロジェクトを研究し、ファイル構造だけでなく**設計意図**を理解したい
- 自分のプロジェクト設計のために、どのパターンが借用可能で、どれを避けるべきかを判断したい
- チームのオンボーディングや知識アーカイブのために構造化されたアーキテクチャレポートを生成したい
- ソースコード証拠に基づく技術的な深掘り記事を書きたい
- リポジトリ概要を作成し、素早くリーディングマップを構築したい
- 複数のリファレンスリポジトリの設計思想を比較したい

## 使わない場面

- 単一ファイルの説明やバグ調査
- 通常のコードレビュー
- 呼び出し関係図だけを描き、設計哲学に関心がない
- 全アーキテクチャのコードマッピングや網羅的な関数分析

## インストール

### 推奨（インタラクティブ）

```bash
npx arch-insight install-release
```

### プラットフォーム指定（非インタラクティブ / CI）

```bash
npx arch-insight install-release --platform codex --platform claude
```

### アップデート

```bash
npx arch-insight update --platform codex
```

| プラットフォーム | インストールパス                              |
| ---------------- | --------------------------------------------- |
| Claude Code      | 自動マーケットプレイス登録                    |
| Codex            | `~/.codex/skills/arch-insight/`               |
| Gemini           | `~/.gemini/skills/arch-insight/`              |
| OpenCode         | `~/.config/opencode/skills/arch-insight/`     |
| Pi               | `~/.pi/agent/skills/arch-insight/`            |
| Kiro             | `~/.kiro/skills/arch-insight/`                |
| Cursor           | `~/.cursor/plugins/local/arch-insight/`       |

インストール後、各プラットフォームを再起動してください。

## 使用方法

**明示的な呼び出し：**

```txt
arch-insight スキルを使ってこのリポジトリを分析し、アーキテクチャドキュメントを生成してください。
```

**自然言語での呼び出し：**

```txt
このプロジェクトのコア抽象と設計トレードオフを研究して、自分のアーキテクチャ設計の参考にしたい
```

```txt
Generate a narrative article about the architecture of this codebase.
```

```txt
このリポジトリのソースコード概要を教えて、リーディングマップを素早く構築したい
```

Skill はユーザーの言葉遣いに基づいて自動的に出力形式を判断します：

- 「ブログスタイル」/「公開可能な長文」/「あの記事のように」 → **ナラティブ記事**
- 「overview」/「リポジトリツアー」/「ソースマップ」 → **リポジトリ概要**
- 「メインレポート + 付録」/「テンプレート出力」 → **分析パッケージ**
- 不明確 → 最小限の推論、分析パッケージにはデフォルトしない

## 出力物

### 分析パッケージ

| ファイル                   | 回答する質問                                    |
| -------------------------- | ----------------------------------------------- |
| `ARCHITECTURE_REPORT.md`   | システム概要と全体判断                          |
| `DESIGN_PHILOSOPHY.md`     | 作者が一貫して堅持している設計原則              |
| `CORE_ABSTRACTIONS.md`     | このシステムの「言語」は何か                    |
| `MAIN_FLOW.md`             | どのパスがアーキテクチャの意図を最も明らかにするか |
| `TRADEOFFS.md`             | 利益、コスト、代替案、境界                      |
| `BORROWABLE_PATTERNS.md`   | 他のチームが実際に持ち帰れるもの                |

### 記事モード

- **ナラティブ記事** → 単一 `NARRATIVE_ARTICLE.md` — 意見があり、設計にフォーカスし、移行コストを考慮
- **リポジトリ概要** → 単一 `REPO_OVERVIEW_ARTICLE.md` — 情報密度が高く、スキャン可能、ソースパスとリーディングナビゲーション付き

## ワークフロー

Skill は段階的なパイプラインに従います：

### 分析パッケージ

1. **Repo Intake** — 範囲、プロジェクトタイプ、ソースとバージョンアンカーを決定
2. **Design Philosophy Brain Dump** — システムメンタルモデル、コア抽象、設計原則の仮説
3. **Ecosystem Atlas** *(オンデマンド)* — monorepo / マルチサービス / プラットフォームシナリオのみ
4. **Architecture Report** — メインレポート + 5 つの学習付録

### ナラティブ記事

1. **Repo Intake**
2. **Design Philosophy Brain Dump**
3. **Narrative Article** — 意見、設計の教訓、移行判断を含む単一の長文

### リポジトリ概要

1. **Repo Intake**
2. **Design Philosophy Brain Dump**
3. **Repo Overview Article** — 情報密度が高く、構造が明確で、スキャン可能、ソースパスとリーディングナビゲーション付き

スタイルサンプルが提供された場合、Agent は執筆前に**スタイル契約**（対象読者、トーン、構造密度、証拠形式、禁止事項）を確立します。契約はテンプレートのデフォルトよりも優先されます。

## 設計原則

- **What より先に Why を問う** — 問題を理解してから構造を説明する
- **ディレクトリツリーよりメインフロー** — 定義的なパスはフォルダレイアウトより価値がある
- **実装詳細よりコア抽象** — 言語を識別してから文法を読む
- **証拠優先** — すべての判断にコードパスを引用する
- **トレードオフを正直に** — すべての設計結論が利益、コスト、代替案、境界をカバーする
- **出力は賢く、長くなく** — 読んだ後に学びがあり、単により多くの文字を見るだけではない
- **網羅ではなく焦点** — 設計哲学と借用可能なパターンが全アーキテクチャコードマッピングより優先される

## 制限事項

- 大規模な monorepo では重要なディレクトリにフォーカスした分析が必要な場合がある
- ランタイム動作（並行モデル、実際のデータフロー）にはログ、テスト、トレーシングが必要 — 静的解析だけでは不十分
- 動的インポート、リフレクション、コード生成、フレームワークマジックは分析精度を低下させる
- 生成されたレポートはプロジェクトメンテナによるレビューが必要
- マルチリポジトリ入力はデフォルトで比較設計リファレンスであり、全エコシステム監査ではない
