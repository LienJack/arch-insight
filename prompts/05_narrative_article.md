# 05 Narrative Article Prompt

你现在要交付的是一篇“可直接阅读和发布”的技术叙事文章，而不是模板化分析包。

## 阶段目标

1. 用开篇 2-4 段说明“这个仓库为什么值得读”。
2. 给出清晰主线：问题定义 -> 架构意图 -> 关键机制 -> 设计取舍 -> 风险判断 -> 总体评价。
3. 用真实代码路径支撑关键判断，但避免目录复述和逐文件解说。
4. 保持批判性：至少指出 1-3 个真实风险或边界条件。

## 强制约束

- 不要输出“主报告 + 五附件”结构。
- 不要写成 checklist 或模板填空。
- 不要按目录树顺序复读代码。
- 不要只有赞美，必须有代价与边界。

## 风格契约（写作前先确认）

在正文前先形成并遵守：

- 受众：工程师 / TL / 架构师
- 语气：克制、清晰、有判断
- 密度：每节有观点，不堆术语
- 证据：关键判断附代码路径
- 禁止项：模板腔、流水账、README 改写

## 推荐结构

1. 开场：为什么这个项目值得研究
2. 项目真实定位（不是官方 slogan）
3. 架构主线与控制权流动
4. 最重要的 2-5 个机制
5. 设计取舍（收益/代价/边界）
6. 风险与改进建议
7. 总体评价（一句有力度的结论）

## 质量闸门

提交前自检：

1. 开头是否回答“为什么值得读”？
2. 是否至少讲透一条主流程？
3. 是否至少有 2 个取舍分析？
4. 是否至少有 1 个批判性判断？
5. 每个重要判断是否有路径依据？

## 产物要求

- 默认输出：`outputs/NARRATIVE_ARTICLE.md`
- 可选附录：`outputs/NARRATIVE_EVIDENCE_INDEX.md`
